# ゲームの出欠をとるアプリケーション 要求仕様

毎週金曜深夜に開催する「桃鉄1年勝負」の出欠を、Discord Bot を用いて自動化するアプリケーションの要求仕様を定める。

> 原文の要求（概要）
> 毎週開催している深夜の桃鉄1年勝負の出欠をとりたい。Discord 上で固定4名の出欠をとり、全員出席なら開催、誰か欠席 / 22時前に未返答があれば中止し、翌日順延の可否を確認する。
>
> ※ 原文の「22時前」は発案時の粗い表現であり、確定した締切は §3.1 の「金 21:30 JST」を正とする。

本書は **業務要求（What / Why）の正典** である。実装手段・技術選定・運用手順・コーディング規約は本書の対象外とする。

---

## 1. 概要

- **対象イベント**: 毎週金曜日深夜の桃鉄1年勝負
- **対象メンバー（固定4名）**:
  - いーゆー (`e.u_ddr`)
  - おーたか (`o_taka1986`)
  - あかねまみ (`akanema3`)
  - ぽんた (`ponta2d`)
- **基準タイムゾーン**: JST (Asia/Tokyo)。本書のすべての時刻表記は特記無き限り JST。
- **運用チャンネル**: 専用チャンネル（例: `#桃鉄出欠`）1 箇所。Bot のすべての投稿・インタラクションはこのチャンネルに限定する。
- **固定4名以外の操作**: 無視する（コマンド実行・ボタン押下ともに）。

---

## 2. 全体フロー

```
[金曜 08:00] 自動で出欠確認メッセージを送信（/ask でも手動起動可）
        │
        │ 4名が5ボタン [22:00][22:30][23:00][23:30][欠席] のいずれかを押す
        │ ※ 締切 21:30 までは回答変更可能
        │
        ├─ 誰かが [欠席] を押下 ─────────→ 即時中止 ─→ 順延確認フローへ
        │
        ├─ 締切 21:30 時点で未回答者あり ──→ 中止確定 ─→ 順延確認フローへ
        │
        └─ 締切 21:30 時点で 4名全員が時刻選択 ─→ 開催確定フローへ
```

---

## 3. 出欠確認（募集）フェーズ

### 3.1 送信タイミング
- **自動送信**: 毎週金曜 **08:00 JST**（開催候補日当日の朝）
  - 順延中は、順延先の当日（土曜）**08:00 JST** に送信
> ※ 順延後（土曜回）の送信タイミングは ADR-0019 により「金曜 `POSTPONED` 成立直後（即時生成・即時送信）」へ更新済み。  
> 送信時刻の正本は `docs/adr/0019-postpone-voting-and-saturday-reask-flow.md` を参照。
- **手動送信**: スラッシュコマンド `/ask`
  - 実行権限: 固定4名のみ
  - **制約**:
    - 同一週内に進行中の募集がある場合は「既に募集中です」と実行者のみに見える応答で返し、**新規募集は作成しない**
    - 順延確認フェーズ中は実行不可
    - **21:30 以降は新規募集を作成しない**（当日分はもう間に合わないため）

### 3.2 募集メッセージ構成
- **メンション**: 固定4名全員
- **本文**（例）:
  ```
  🎲 今週の桃鉄1年勝負の出欠確認です

  開催候補日: 2026-04-24(金) 22:00 以降
  回答締切:   21:30（未回答者が残っていれば中止）
  ルール:     「欠席」が1人でも出た時点で中止 / 押した時刻 "以降" なら参加OK
              （例: 23:00 を押すと 23:00/23:30 でも参加可能として集計されます）

  【回答状況】
  - いーゆー    : 未回答
  - おーたか    : 未回答
  - あかねまみ  : 未回答
  - ぽんた      : 未回答
  ```
- **ボタン（横1列・5個）**: `22:00` / `22:30` / `23:00` / `23:30` / `欠席`

### 3.3 回答挙動
- 4名それぞれが5ボタンのいずれか1つを押して回答する。
- 回答は **募集メッセージ本文を編集して全員に可視化**する（各メンバーの選択 or 「未回答」）。
- **回答の変更**: 締切（21:30）までは自由に押し直し可能。最後に押したボタンを有効とする。
- **中止確定後**（欠席押下 or 締切未回答）は全ボタンを無効化し、以降の変更は不可。

## 4. 判定ロジック

### 4.1 中止確定（以下のいずれかで中止確定）
| 条件 | 発動タイミング |
|---|---|
| 誰か1人が `欠席` を押下 | 押された瞬間に即時 |
| 締切（金 21:30）時点で未回答者が1人以上残存 | 締切到達時 |

中止確定後は §6 の順延確認フローへ遷移する。

### 4.2 開催確定
- **締切（21:30）時点で 4名全員が時刻ボタンを選択している**ことが条件。
- **集計ルール（「それ以降参加可能」解釈）**:
  - 各ボタンの意味: 「その時刻以降に参加可能」
  - 22:00 選択者は 22:00 / 22:30 / 23:00 / 23:30 すべてで参加可能
  - 23:00 選択者は 23:00 / 23:30 のみ参加可能
  - **開始時刻 = 4名の選択時刻のうち最遅のもの**（= 全員が参加可能な最早時刻）
- タイブレーク不要（開始時刻は常に一意に確定する）。
- 締切前に全員が時刻選択を終えていても、開催確定は **締切時点** で行う（それまでは暫定表示）。

### 4.3 暫定状態の表示
全員が時刻選択済みだが締切未到達の間は、募集メッセージの末尾に暫定開始時刻を表示する：
```
暫定開始時刻: 23:00（21:30 の締切で確定）
```

## 5. 開催確定フェーズ

### 5.1 開催決定メッセージ
- **送信タイミング**: 締切（21:30）到達時、4名全員が時刻選択済みだった場合に即時投稿
- **メンション**: 参加者（＝ 4名）
- **本文**（例）:
  ```
  🎉 今週の桃鉄1年勝負は開催します！

  開始時刻: 23:00
  回答内訳:
  - いーゆー    : 22:30
  - おーたか    : 23:00
  - あかねまみ  : 22:00
  - ぽんた      : 22:30
  ```

### 5.2 15分前リマインド
- **送信タイミング**: 開始時刻の 15 分前
- **メンション**: 参加者（＝ 4名）
- **本文**（例）:
  ```
  ⏰ 15分後に開始です（23:00 開始）
  ```
- **取りこぼし対策**: 開催確定からリマインド予定時刻まで10分未満の場合、**リマインドは送らない**（直前メッセージとなり冗長のため）。

## 6. 中止・順延フェーズ

### 6.1 中止メッセージ
- **送信タイミング**: 中止確定の瞬間
- **メンション**: 固定4名全員
- **本文**（例）:
  - 欠席による中止:
    ```
    ❌ 今週の桃鉄1年勝負は中止です
    理由: おーたか さんが欠席を選択したため
    続けて、明日への順延確認を行います。
    ```
  - 未回答による中止:
    ```
    ❌ 今週の桃鉄1年勝負は中止です
    理由: 21:30 の締切時点で未回答者がいたため（未回答: いーゆー / あかねまみ）
    続けて、明日への順延確認を行います。
    ```

### 6.2 順延確認メッセージ
- **送信タイミング**: **金曜日の中止時のみ**、中止メッセージの直後に別メッセージで送信
  - **土曜日（順延先）の中止時は順延確認を行わない**（§6.3）
- **メンション**: 固定4名全員
- **本文**（例）:
  ```
  📅 明日（土曜）に順延しますか？

  「明日の募集OK」= 明日もう一度出欠を取ってよい、という意思表示です。
  （※ この時点では参加確定にはなりません）

  全員が【明日の募集OK】を押したら、土曜回の出欠確認を送ります。
  1人でも【今週は見送り】を押したら、その週は終了です。
  回答期限: 本日 24:00（= 翌日 00:00 JST）

  【回答状況】
  - いーゆー    : 未回答
  - おーたか    : 未回答
  - あかねまみ  : 未回答
  - ぽんた      : 未回答
  ```
- **ボタン**: `明日の募集OK` / `今週は見送り`
- **判定**:
  - 4名全員が `明日の募集OK` を押した → `POSTPONED` 成立直後に、翌日（土曜）回の出欠確認を作成・送信（詳細は ADR-0019）
  - 1人でも `今週は見送り` を押した → その週は終了（「今週の桃鉄は見送りです」を投稿）
  - 本日 24:00 までに4名全員の回答が揃わなかった → その週は終了

### 6.3 順延の制約
- **順延は1回のみ**（金曜→土曜の1段階）。土曜日に中止となった場合は **順延確認を出さず**、その週は完全終了。
- 順延後（土曜日）の運用は、送信時刻・締切・集計ルールとも金曜日と同じ（締切 土 21:30、22:00 以降開催）。
> ※ 送信時刻のみ ADR-0019 により「金曜 `POSTPONED` 成立直後」に置換。締切・集計ルールは本節のとおり。

### 6.4 順延期限の表記
- 「本日 24:00」は **候補日翌日 00:00 JST** を指す慣習表記であり、仕様上サポートするのはこの表記のみ。`25:00` などの 24 超え表記は受け付けない。

## 7. 管理コマンド（スラッシュコマンド）

全て実行権限は固定4名のみ。

| コマンド | 機能 |
|---|---|
| `/ask` | 手動で出欠確認を起動（§3.1 の制約あり） |
| `/cancel_week` | その週の運用を緊急スキップ |
| `/status` | 現在の週の募集状況を実行者のみに見える応答で表示 |

### `/cancel_week` の副作用
- 現在週の状態を `SKIPPED` に遷移
- 進行中の募集メッセージ・順延確認メッセージの**全ボタンを無効化**
- 未送信のリマインド・自動再送予定を取消
- チャンネルに「今週は運用都合により見送りです」を投稿（実行者名を含める）
- 実行時は実行者のみに見える応答で「本当に今週の運用をスキップしますか？」と**確認ダイアログ**を挟む

## 8. データ永続化（論理モデル）

Bot 再起動時の状態復元と重複実行防止のために、以下の論理モデルを永続化する。物理スキーマ・テーブル名・カラム名・整合制約の実装手段は要求仕様の対象外とする。

### 8.1 Session（週次募集本体）
1 週・1 順延回数ごとに 1 件。以下の情報を保持する：

- **週キー**: ISO week 表記（開催候補日の金曜基準）
- **開催候補日**（JST）
- **順延回数**: 0（金曜回）または 1（土曜への順延回）
- **状態**: `ASKING` / `POSTPONE_VOTING` / `POSTPONED` / `DECIDED` / `CANCELLED` / `COMPLETED` / `SKIPPED`
  - 終端状態は `COMPLETED`（通常完了）または `SKIPPED`（`/cancel_week` 実行）
  - `CANCELLED` は中止確定直後の一時状態で、順延判定または週終了処理を経て `COMPLETED` または `POSTPONED` に遷移する
  - `POSTPONE_VOTING`: 金曜 `ASKING` 中止後の順延意思確認フェーズ
  - `POSTPONED`: 順延投票で全員 OK が揃ったことを示す終端（成立直後に土曜 `ASKING` を新規作成）
- **中止・完了の理由**（`cancelReason` 例）
  - `absent_cancelled` / `unanswered_cancelled` / `postpone_ng` / `postpone_unanswered` / `saturday_cancelled`
- **運用チャンネル**および、募集メッセージ・順延確認メッセージを指す識別子（再描画のために必要）
- **締切日時** / **リマインド予定時刻** / **リマインド送信済み時刻**
- **確定した開始時刻**（開催確定時のみ）

**一意性**: 同一の週キー・順延回数の組で進行中の Session は最大 1 件。

### 8.2 Response（各メンバーの回答）
- Session・メンバー・選択肢（`T2200` / `T2230` / `T2300` / `T2330` / `ABSENT` / `POSTPONE_OK` / `POSTPONE_NG`）・回答時刻。
- 同一 Session × メンバーで一意（最新回答で上書き）。

### 8.3 開催履歴（将来の桃鉄戦績集計システムと統合前提）
永続的に保持：
- **HeldEvent**: 実開催日・開始時刻・紐づく Session・参加メンバー一覧。

### 8.4 保存しないもの
- 回答の変更履歴（最新の回答のみ保持）
- 中止回の詳細（Session には残るが `HeldEvent` は作成しない）

## 9. 状態遷移

```
         ┌──────── /ask, 金08:00 ──────┐
         ▼                              │
     [ASKING] ──── 欠席押下 ───→ [CANCELLED] ──┐
         │                                     │
         │ 締切到達                            │
         ├── 未回答あり ────────→ [CANCELLED] ─┤
         │                                     │
         └── 全員時刻選択 ──→ [DECIDED] ──→ [COMPLETED]
                                   │
                                   ▼ 開始時刻到達の15分前
                               リマインド送信（送信時刻を記録）

   [CANCELLED] ┬─ 金曜 ─→ [POSTPONE_VOTING]
               │              │
               │              ├── 全員OK ─→ [POSTPONED] ─→ 翌日 順延回数=1 の新 [ASKING] 生成
               │              ├── 1人NG ─→ [COMPLETED]（理由: 順延 NG）
               │              └── 当日 24:00 未完 ─→ [COMPLETED]（理由: 順延未完）
               │
               └─ 土曜 ─→ [COMPLETED]（理由: 土曜中止。順延確認は出さない）

   /cancel_week は任意時点から [SKIPPED] へ遷移（以降のジョブ・ボタンはすべて無効化）
```

### 9.1 DECIDED から COMPLETED への遷移
- `DECIDED` 状態の Session は、15 分前リマインドの送信処理が完了した時点（リマインド不要条件に該当する場合はその判定時点）で `COMPLETED` に遷移する。
- Bot は実開催そのものの監視・終了判定を行わない。

### 9.2 状態遷移の要求
- **同時操作に耐えること**: 4 名が同時に押しても、外部タイマと同時に発火しても、二重処理・状態の取り違え・回答のロストが発生しない。
- **再起動に耐えること**: Bot が再起動しても、進行中の Session は永続化された状態だけから復元でき、締切・リマインド判定が継続される。
- **`/cancel_week` 最優先**: 任意時点で実行可能で、実行直後に他のすべての自動処理（再描画・締切判定・リマインド送信・自動再送）を停止させる。
- **表示と状態の分離**: 募集・順延確認メッセージの編集失敗は状態遷移を取り消す理由にならない。メッセージは常に永続化された状態から再構築する。

## 10. 決定済み仕様サマリー

| 項目 | 決定内容 |
|---|---|
| 基準タイムゾーン | JST (`Asia/Tokyo`) 固定 |
| 締切 | 金 21:30 JST（順延時は土 21:30） |
| 自動送信 | 金 08:00 JST（順延時の送信タイミングは ADR-0019 に従い、金曜 `POSTPONED` 成立直後に実行） |
| 回答UI | 5ボタン（22:00/22:30/23:00/23:30/欠席）を1メッセージ内に配置 |
| 集計 | 「それ以降参加可能」解釈、開始時刻 = 選択時刻の最遅値 |
| 欠席挙動 | 1人でも欠席→即中止、残り回答打ち切り |
| 回答変更 | 締切まで可、中止確定後は不可 |
| 順延判断 | 4名全員が「明日の募集OK」で翌日再出欠。1人でもNG or 当日 24:00 までに未完なら週終了 |
| 順延期限表記 | `24:00` は「候補日翌日 00:00 JST」として解釈する唯一の慣習表記 |
| 順延回数 | 1回のみ（土曜まで）。土曜中止時は順延確認なし |
| 週キー | ISO week 表記（金曜基準）。金曜 Session（順延 0 回）と土曜 Session（順延 1 回）は同一週キーを共有 |
| メンション | 出欠確認・中止・順延確認 = 4名全員 / 開催決定・15分前リマインド = 参加者（4名） |
| 投稿先 | 専用チャンネル1箇所 |
| コマンド権限 | 固定4名のみ |
| スキップ | `/cancel_week` で緊急スキップ（事前スケジュール機能は不要） |
| 履歴保存 | 開催日・開始時刻を永続保存（将来の戦績集計システムと統合前提） |
