# ADR-0002: JST 固定と時刻処理の集約

## Context

この Bot は日本国内の 1 Guild・1 チャンネル・固定 4 名だけを対象に運用し、開催可否の判定時刻も全て日本時間で説明される。
毎週の募集は金曜 18:00、締切は 21:30、順延確認の期限は「本日 24:00」という日本語表現で扱うため、内部実装の時刻基準が分散すると仕様解釈がぶれやすい。
Date の内部表現が UTC であっても、判定や表示や運用ログが JST 以外で解釈されると、締切判定や順延期限の説明が利用者の認識と一致しなくなる。
金曜と土曜が年を跨ぐ週では、暦年と ISO week year が一致しないことがあり、週キーを自前で組み立てると誤集計や二重作成を起こしやすい。
「24:00」は日本的には自然だが、一般的な時刻パーサでは当日 24 時と翌日 00 時のどちらかが曖昧になりやすく、`25:00` などの拡張表記まで許すと仕様境界が崩れる。
また、cron 判定や締切処理をグローバル時計に直接結びつけると、年跨ぎ・締切直前・順延期限切れなどの重要ケースを安定してテストしにくい。
単一運用である以上、時刻の柔軟性よりも「どこで判定しても同じ結果になる単純さ」を優先する必要がある。

## Decision

- アプリのタイムゾーンは起動時に `process.env.TZ = "Asia/Tokyo"` を設定し、以後の仕様判定は常に JST で行う。
- Date の内部表現は UTC を含め任意とするが、判定・表示・ログ出力で参照する時刻はすべて JST 基準に正規化する。
- DST は考慮しない。対象運用は日本国内に限定し、JST 固定で扱う。
- 現在時刻取得、週キー算出、候補日計算、締切計算、リマインド予定算出、`POSTPONE_DEADLINE` 解釈は `src/time/` に集約する。
- `src/time/` 以外で `new Date()`、`Date.parse()`、日付文字列の手組み連結による仕様判定を禁止する。
- 週キーは金曜回を基準に、`date-fns/getISOWeek` と `getISOWeekYear` を併用して算出する。
- 金曜 Session（順延回数 0）と土曜 Session（順延回数 1）は同一の週キーを共有する。
- `YYYY-Www` のような週キー文字列を暦年ベースで自作しない。
- `POSTPONE_DEADLINE="24:00"` は「候補日翌日 00:00 JST」のみとして解釈する。
- `25:00` などの 24 超え表記は parse 段階で拒否し、受け入れない。
- 現在時刻は fake timers または依存性注入で差し替え可能にし、時刻依存ロジックをグローバル時計へ直結させない。

## Consequences

### Positive

- 締切、順延期限、募集送信、リマインドがすべて利用者の認識する JST と一致し、説明と実装のずれが減る。
- 時刻計算の入口が `src/time/` に一本化されるため、レビュー対象が明確になり、`new Date()` の散在による不整合を抑制できる。
- 金曜 12/31 と土曜 1/1 のような年跨ぎでも、ISO week year を正しく使って同一週の Session を識別できる。
- `24:00` の解釈を一意に固定することで、順延期限表示と内部判定が一致する。
- fake timers や DI を使って締切直前、締切直後、順延期限切れ、年跨ぎのテストを安定して再現できる。
- ログや障害調査でも JST のまま時系列を追えるため、運用時の読替えコストが下がる。

### Negative

- 複数タイムゾーン対応や海外展開には向かず、将来それが必要になった場合は別の ADR で前提を見直す必要がある。
- 時刻処理を `src/time/` に寄せるぶん、簡単な日時参照でも専用 API を経由する設計コストが増える。
- 一般的な「文字列をそのまま Date に渡す」実装より厳格な parse ルールを持つため、入力値の制約を保守し続ける必要がある。
- `src/time/` の API 設計が不十分だと、呼び出し側が回避策を作りたくなるため、継続的なレビューが必要になる。

## Alternatives considered

- **UTC 保持 + 表示時だけ JST 変換**: 保存方法としては可能だが、締切判定や順延期限判定まで UTC 発想で書く必要があり、業務時刻の読替えミスを誘発するため採用しない。
- **複数 IANA タイムゾーン対応**: 利用者も運用 Guild も日本国内固定であり、用途に対して複雑さが過剰なため採用しない。
- **`Intl.DateTimeFormat` ベースで週キーを組み立てる**: ISO week / ISO week year を直接安全に扱いにくく、年跨ぎ条件の実装が複雑になるため採用しない。
- **`24:00` を「当日終端の 24 時」として緩く扱う**: 翌日 00:00 との境界が曖昧になり、`25:00` などの拡張表記の扱いもぶれるため採用しない。
