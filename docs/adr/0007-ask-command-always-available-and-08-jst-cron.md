# ADR-0007: `/ask` コマンドの常時実行許可と自動送信時刻 08:00 JST

## Context
- `requirements/base.md` は自動投稿を「金曜 18:00 JST」と定めていたが、運用者の生活リズムに合わせ、本人の起床前後に確認できる時間帯へ移したい要望が出た。
- 自動送信のみに依存すると、テスト投稿・障害復旧後の再投稿・突発的な募集やり直しに手段がなくなる。`/ask` を運用者手動で随時発火できることが必要になった。
- 一方で `/ask` に曜日・時刻ガードを入れると「テストしたいのに当日でないと打てない」「順延候補日と現在時刻の整合確認が必要」など、ロジックが二箇所に分散して複雑化する。
- 当プロジェクトは固定 4 名の Bot で不特定多数には公開されないため、手動コマンドの乱用リスクは極めて低く、むしろ手動経路の自由度を確保するメリットが大きい。

## Decision
- 自動送信スケジュールを **毎週金曜 08:00 JST** に固定する。cron 式は `0 8 * * 5`（timezone `Asia/Tokyo`、多重起動防止付き）。
- `/ask` スラッシュコマンドは **曜日・時刻の制限を設けず、常時実行可能**とする。
  - ただし Interaction 検証（guild / channel / member）と週次重複防止（同一 ISO 週に 1 通）は常に適用する。
- 手動 `/ask` と cron 自動送信が同一週に重複しないよう、送信経路は共通の `sendAskMessage` に集約し、週キー単位で冪等化する（cron と `/ask` のどちらが先でも、2 通目はスキップされる）。
- 送信時点の「開催候補日」は `src/time/candidateDateForSend` の結果に従う。非金曜実行時の文言ルールは未確定のため、仕様確定まで現在日時をそのまま候補日として提示する（`// TODO(ai): spec clarification needed` 相当の扱い）。

## Consequences
- Positive
  - 自動送信失敗・Discord 障害・テスト用途でも運用者が随時 `/ask` でリカバリできる。
  - 自動送信時刻が朝にずれたことで、運用者が日中にリマインドや調整を行いやすい。
  - ガード条件を週キー重複防止に一本化することで、手動経路と自動経路の挙動差分を抑えられる。
- Negative
  - `requirements/base.md` の「18:00 自動送信」記述と乖離する。仕様書側の更新が必要になる。
  - 非金曜の `/ask` 実行では「開催候補日」の文言が運用者の期待とずれる可能性がある（仕様未確定）。
  - 手動実行を誰でも（メンバー 4 名のいずれでも）打てるため、誤爆時は週次重複防止に頼る形になる。
- Operational implications
  - cron の tick 時刻変更（18:00 → 08:00）は `src/scheduler/` にのみ影響する。
  - 「デプロイ禁止窓（金 17:30〜土 01:00 JST）」とは別枠で、朝 08:00 の自動送信が走ることを運用者は認識しておく必要がある。
  - 非金曜の表示文言仕様が確定した時点で ADR を追加または更新する。

## Alternatives considered
- **`/ask` に曜日・時刻ガードを設ける**
  - 却下理由: 運用者手動リカバリの唯一の手段を塞ぐことになり、テスト・障害復旧で手詰まりになる。固定 4 名の Bot で乱用リスクも低い。
- **自動送信時刻を 18:00 のまま維持**
  - 却下理由: 運用者の生活リズムと合わず、朝のうちに確認・調整したいという実運用要望を満たせない。
- **手動 `/ask` を自動送信とは別のチャンネルで提供**
  - 却下理由: `env.DISCORD_CHANNEL_ID` 1 チャンネル運用の原則に反し、設定・権限が複雑化する。
