# ADR-0006: ドキュメント構造（業務要求 / 運用 / レビュー規約の分離）

## Context

このプロジェクトは人間開発者だけでなく、Copilot、Claude Code、Codex など複数の AI エージェントも参照しながら保守される。
過去の `requirements/base.md` は約 680 行規模に膨らみ、業務要求、実装方針、レビュー観点、運用手順が同居していたため、どれが正本か判断しにくかった。
業務要求と実装技術の記述が同じ文書に混ざると、仕様変更と実装都合の差分が区別しづらくなり、重複転記による齟齬も起きやすい。
Copilot Code Review は instructions ファイルを先頭側から優先的に読む前提があり、参照チェーンに頼る設計では必要な規約が読み落とされやすい。
一方で、人間向け README にはセットアップや運用の入口が必要であり、AI 向け作業手順やレビュー規約とは期待される情報密度が異なる。
判断の根拠を PR 本文だけに残す運用では、時間経過後に「なぜそう分割したのか」を再検索しづらく、文書構造そのものの変更理由が失われやすい。
文書の役割を分けないまま拡張すると、同じ事実を複数箇所へ転記し続ける構造になり、更新漏れが常態化する。

## Decision

- `requirements/base.md` は業務要求仕様だけを保持し、What / Why に集中させる。
- `requirements/base.md` には実装技術名、採用ライブラリ、コード例、ローカル開発手順、デプロイ手順、運用ノートを持ち込まない。
- `README.md` は人間向けの入口とし、技術スタック、環境変数、ローカル開発、CI/CD、デプロイ運用、死活監視、脆弱性対応、将来拡張を集約する。
- `.github/instructions/*.md` は applyTo ベースで自動適用されるレビュー規約として扱う。
- instructions は自己完結にし、他文書を読まなくてもレビュー判断に必要なルールを先頭側で理解できるようにする。
- instructions の単位は `runtime`、`interaction-review`、`db-review`、`time-review`、`secrets-review` の 5 つとする。
- `AGENTS.md` は AI エージェント向けナビゲーションとし、SSoT 表、作業手順、禁止領域、既知の落とし穴をまとめる。
- `.github/copilot-instructions.md` は常時注入される要約ルールとし、判断に必要な要約を含めつつ詳細は各正本に委ねる。
- `docs/adr/` に ADR を保管し、各決定の理由、比較した代替案、採択時点の前提を残す。
- ルール本文は AGENTS や instructions に置き、なぜその構造にしたかの説明責任は ADR に置く。

## Consequences

### Positive

- 業務要求、運用手順、レビュー規約、AI 作業ナビゲーションの責務が分離され、どの文書を更新すべきか判断しやすくなる。
- 業務仕様変更と実装都合の変更が別文書になるため、差分レビューで「仕様が変わったのか」「実装ルールが変わったのか」を識別しやすい。
- instructions を自己完結させることで、Code Review が参照チェーンを辿れなくても重要ルールを読み落としにくくなる。
- AI エージェントが混在しても SSoT を明示でき、同じ内容の重複記述とズレを減らせる。
- ADR に理由を集約することで、将来の文書再編時にも判断根拠を検索しやすい。
- 人間向け README の可読性を保ちながら、AI 向け規約は applyTo 単位で必要箇所だけを読み込ませやすくなる。

### Negative

- 文書数が増えるため、変更時には「どの層の文書を触るべきか」を意識して運用する必要がある。
- 同じテーマが複数文書にまたがる場合、責務境界を守らないと再び重複が発生する。
- instructions を自己完結に保つため、短い同種ルールを複数ファイルへ再記述するコストが一部発生する。
- 文書再編の初期段階では、既存記述をどこへ移すべきかの棚卸しコストが発生する。
- 各文書の責務をレビューで継続的に監視しないと、時間とともに再び混在へ戻る可能性がある。

## Alternatives considered

- **単一の `base.md` にすべて集約する**: 文書が肥大化し、Code Review で必要箇所を読み切れず、業務要求と実装・運用の境界も曖昧になるため採用しない。
- **全 AI エージェント向け instruction を 1 ファイルへ集約する**: applyTo の粒度が失われ、変更領域ごとのレビュー規約を自動適用しづらくなるため採用しない。
- **判断の根拠を PR 本文だけに残す**: 永続性と検索性が弱く、後から構造変更の理由を再利用しにくいため採用しない。
- **README を AI 向け規約の正本にする**: 人間向け入口の役割と衝突し、セットアップ案内とレビュー規約が混在して可読性が落ちるため採用しない。
